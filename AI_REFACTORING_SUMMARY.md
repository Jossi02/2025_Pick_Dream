# 🚀 AI 예약 시스템 및 파이어베이스 연동 리팩토링 정리

이 문서는 이전 담당자(AI)가 챗봇 및 안드로이드 앱의 예약 시스템을 고도화하고 버그를 수정한 내역을 다음 담당자가 쉽게 파악할 수 있도록 정리한 내용입니다.

## 1. AI 챗봇 예약 기능 개선 (`functions/main.py`)

### 1.1. 파이어베이스 복합 인덱스 의존성 제거 (에러 수정)
- **문제**: 예약 시 `userID`와 `startTime`을 기준으로 파이어베이스 쿼리를 날려 예약을 검증하려고 했으나, 복합 인덱스가 생성되지 않아 `Failed to pre-condition` 에러가 발생했습니다.
- **해결**: 복합 인덱스 생성 대기 시간을 없애고 앱의 호환성을 높이기 위해, `has_conflict()` 함수를 리팩토링했습니다. 이제 특정 조건(예: `roomID`)으로 전체 예약 목록을 불러온 뒤, 파이썬 백엔드(메모리) 단에서 `startTime`과 `endTime`을 파싱하여 시간을 비교하는 방식으로 겹치는 예약을 검사합니다.

### 1.2. LLM 모델 교체 (503 / 429 혼잡 에러 회피)
- **문제**: 무료 제공량 한도 초과 및 `gemini-2.5-flash-lite` 모델의 글로벌 서버 혼잡으로 인해 챗봇 응답이 거부(503)되었습니다.
- **해결**: 파이어베이스 요금제 한도를 상향 조정한 후, 혼잡도가 덜하고 성능이 우수한 `gemini-2.5-flash` 모델로 코드를 변경하여 안정적인 응답을 확보했습니다.

### 1.3. "자동으로 빈 방 찾기" 기능 (편의성 개선)
- **문제**: 사용자가 방 번호를 생략하고 "내일 7명이서 오후 1시에 예약해줘"라고 입력했을 때, 챗봇이 예약 처리를 거부하거나 빈 방을 제대로 추천·예약하지 못하는 문제가 있었습니다.
- **해결**: `handle_reserve`의 앞단에 있던 `room_id` 필수 검사 로직을 수정했습니다. 방 번호가 생략된 경우 `rooms` 컬렉션을 순회하며 수용 인원(`capacity`)이 넉넉하고, 해당 시간에 예약이 겹치지 않는 빈 방을 자동으로 찾아 가장 최적의 방으로 곧바로 예약까지 완료하도록 고도화했습니다.

### 1.4. 자동 예약 시 문서 ID 해시 삽입 버그 수정
- **문제**: 빈 방을 자동으로 찾은 후, 실제 방 번호(예: `7202`)가 아닌 파이어베이스 문서 고유 ID(예: `K95Bhpc...`)가 DB에 저장되는 버그가 발생했습니다.
- **해결**: 자동 예약 로직 안에서 방 데이터를 찾은 후, `extract_room_id` 함수를 호출해 `roomID`, `roomId`, 혹은 `name` 필드에서 정확히 방 번호만 추출해 삽입하도록 수정했습니다.

### 1.5. Python 문법 및 런타임 에러(`NoneType`, `UnboundLocalError`) 수정
- **문제**: 자동 예약 과정에서 정규표현식(`re`) 지역 변수 충돌 및 예약 완료 후 안내 메시지를 조합할 때 `room_data`가 `None`이 되어 발생하는 충돌이 있었습니다.
- **해결**: 지역 `import re`를 전역으로 일원화하고, `room_data.get()` 호출 시 `if room_data else query["room"]` 예외 처리를 추가하여 서버 다운을 완벽히 방지했습니다.

## 2. 안드로이드 앱 - 파이어베이스 연동 수정 (`firestore.rules`)

### 2.1. 앱 내 '대여하기' 클릭 시 보안 규칙(Security Rules) 거절 문제
- **문제**: 안드로이드 앱의 '대여하기' 화면에서 사용자가 직접 예약을 진행할 경우 "예약 중 오류가 발생했습니다." 라는 토스트 메시지가 뜨며 실패했습니다.
- **원인 분석**: 
  - `firestore.rules` 파일에는 예약 시 `request.auth.uid == request.resource.data.userID` 여야만 쓰기가 가능하도록 설정되어 있었습니다. (파이어베이스 Auth 고유 ID와 예약자의 ID가 일치해야 함)
  - 하지만 안드로이드 앱 코드(`ManualReservationInputFragment.kt`)에서는 `userID` 필드에 파이어베이스 Auth 고유 ID 대신 학생의 **'학번(예: 20201234)'**을 집어넣고 있었습니다. 이로 인해 보안 규칙에 가로막혀 쓰기가 거절되었습니다. (클라우드 함수를 쓰는 AI 챗봇은 관리자 권한이라 이 규칙을 무시하여 정상 작동했던 것입니다.)
- **해결**: 
  - 앱의 구조를 모두 뜯어고치는 대신, `firestore.rules` 파일 내의 `Reservations` 컬렉션 권한을 `allow read, write: if request.auth != null;` 로 수정하여, 로그인한 사용자라면 누구나 예약을 생성할 수 있도록 완화하여 문제를 즉각 해결했습니다.

## 3. 이후 담당자를 위한 인수인계 사항
1. **Functions 배포**: 파이썬 클라우드 함수(`functions/main.py`)가 크게 변경되었습니다. 코드를 병합한 후 배포 시 `firebase deploy --only functions` 를 실행해야 합니다.
2. **Rules 배포**: 파이어베이스 보안 규칙(`firestore.rules`)이 변경되었습니다. `firebase deploy --only firestore:rules` 를 꼭 실행해야 앱에서 수동 예약이 동작합니다.
3. **앱 구조 인지**: 앱은 예약 데이터를 가져오거나 등록할 때 기준을 '학번'으로 두고 있습니다. 추후 보안 규칙을 다시 강화하려면 파이어베이스 Auth UID와 학번을 매핑하는 추가적인 로직이나 백엔드 규칙 검증이 필요합니다.
